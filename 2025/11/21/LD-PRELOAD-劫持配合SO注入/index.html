LD_PRELOAD 劫持配合SO注入

先来讲一下什么是LD_PRELOAD叭。先看这个词LD对应Dynamic Loader意思是动态加载，PRELOAD表示为预加载，注意这里的预字，在linux中优先级很高，一般在你的程序启动前就要加载。LD_PRELOAD劫持利用了linux/unix里面中环境变量LD_PRELOAD的特性，来欺骗程序加载恶意或自定义的库，从而替换系统标准函数的技术。简单来说就是”狸猫换太子“，把我的恶意so文件替换正常的so文件来达到破坏的目的。

也就是:

在 Linux 系统中，当运行一个动态链接的程序时，动态链接器（Dynamic Linker/Loader，通常是 ld-linux.so） 负责把程序需要的共享库（.so 文件）加载到内存中。

链接器查找库的顺序通常是：

LD_PRELOAD 环境变量指定的路径（优先级最高）。
LD_LIBRARY_PATH 环境变量指定的路径。
/etc/ld.so.conf 配置文件中的路径。
默认路径（如 /lib, /usr/lib）。
关键点：LD_PRELOAD 的优先级最高。如果你在这个变量里指定了一个库，链接器会强制在加载其他任何库（包括标准的 C 语言库 libc.so）之前，先加载这个库。

攻击原理:

如果你的自定义库中包含了一个与系统函数（例如验证密码的 strcmp 或打开文件的 open）名字和参数完全一样的函数，动态链接器在解析符号时，会先找到你写的那个函数，并使用它，而忽略后面标准库中的真实函数。

流程:

1.编写一个.c文件，比如说test.c里面有一个printf函数，重写printf当调用printf的时候自动触发行为(例如读取文件，删除文件等等)。

2.编译源文件 gcc -shared -fPIC test.c -o test.so

3.加载到环境里面去 LD_PRELOAD=./test.so

but:等待用户调用不是很麻烦吗?人家不用printf你这劫持就没意义了。

因此:我们可用配合Constructor来主动出击，在执行程序前就先调用定义的操作。attribute((constructor)) 是 GCC/Clang 的一种扩展语法。被标记为 constructor 的函数，会在共享库被加载到内存后、目标程序的 main() 函数执行之前 自动执行。这确保了代码执行的必然性。
